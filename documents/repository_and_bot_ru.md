# Объяснение репозитория Docker-Claw

## Что это за проект
`Docker-Claw` — Node.js/Express сервер, который под каждого пользователя поднимает отдельный Docker-контейнер (sandbox), дает API для управления файлами/командами/БД и подключает AI-агента для автоматического выполнения задач.

Ключевая идея: один `chat_id` = одна изолированная среда с отдельными файлами и PostgreSQL-базой.

## Основные части проекта
- `server.js` — точка входа. Инициализирует storage/snapshots, восстанавливает сессии, запускает Telegram и Email каналы, подключает маршруты API.
- `routes/` — REST API:
  - `/api/session` — создание/статус/удаление сессий;
  - `/api/execute` — выполнение команд в контейнере;
  - `/api/files`, `/api/database`, `/api/apps`, `/api/plans` — операции с файлами/БД/приложениями/планами;
  - `/api/manage` — управление Telegram/Email/AI настройками.
- `services/session.service.js` — жизненный цикл пользовательской сессии: создать/восстановить/удалить контейнер, выполнить команду, чистка idle-сессий.
- `services/docker.service.js` — низкоуровневая работа с Docker (`run/exec/cp/rm`), запуск контейнера, выполнение команд, базовые проверки опасных команд.
- `services/storage.service.js` — директории данных, бэкапы, очистка старых бэкапов.
- `manage/store.js` — персистентное состояние по `chat_id` (токены, AI-настройки, история сообщений, контекст).
- `manage/telegram/*` — Telegram-бот, агентный цикл и обработчики инструментов.
- `manage/email/processor.js` — опрос IMAP, обработка писем и ответы через SMTP.
- `routes/webhook.routes.js` — API-вебхук для внешнего вызова AI-агента.
- `routes/user_hooks.routes.js` — пользовательские хуки `/hook/:chatId/*`, которые запускают `webhook_handler.js` или `webhook_handler.py` внутри контейнера.

## Как работает бот (по шагам)
1. Пользователь пишет в Telegram или Email (либо приходит webhook).
2. Для `chat_id` поднимается/находится сессия: отдельный контейнер `sandbox-user-{chatId}`.
3. Собирается контекст (история диалога, состояние проекта, системные инструкции).
4. Запрос отправляется в AI Router (`services/ai_router_service.js`) с выбранной моделью и списком tools.
5. Модель возвращает либо текст, либо tool-calls.
6. `manage/telegram/toolHandlers.js` исполняет инструменты:
   - чтение/запись/патч файлов;
   - запуск bash-команд в контейнере;
   - создание Node.js приложений;
   - запуск тестов;
   - HTTP-запросы и т.д.
7. Цикл повторяется, пока агент не вызовет `task_completed` с финальным результатом.
8. Ответ и файлы отправляются обратно в канал (Telegram/Email/Webhook), история сохраняется в `manage-store`.

## Важные механизмы
- Очередь задач per `chat_id` (`manage/agentQueue.js`) не дает одновременно выполнять конфликтующие запросы в одной сессии.
- Восстановление после перезапуска: сервер сканирует существующие контейнеры и поднимает их в память.
- Регулярные процессы: очистка idle-сессий, бэкапы, очистка старых snapshot/backup.
- Верификация Telegram-пользователя через одноразовый код.
- Защита в HTTP tool от SSRF на приватные IP.

## Коротко про режимы агента
В проекте есть наборы инструментов для режимов:
- `CHAT` — упор на диалог;
- `WORKSPACE` — работа с файлами/проектом;
- `TERMINAL` — команды и автоматизация внутри контейнера.

Фактически ядро бота — это цикл: **контекст -> LLM -> вызовы инструментов -> проверка результата -> финальный ответ**.
